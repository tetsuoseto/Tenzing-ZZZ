## はじめに

Codexは、OpenAIのSecurity、Product Engineering、Frontend、API、Infrastructure、Performance Engineeringなど、多数の技術チームで日々使用されています。チームはこれを、複雑なシステムの理解や大規模コードベースのリファクタリングから、新機能の提供や厳しい締め切り下でのインシデント対応まで、幅広いエンジニアリング作業を加速するために活用しています。

OpenAIのエンジニアへのインタビューと社内の利用データをもとに、Codexがどのようにチームの作業をより速く進め、作業品質を向上させ、大規模な環境での複雑性を管理するのに役立つかを示すユースケースとベストプラクティスをまとめました。

## ユースケース 1: コード理解

Codexは、オンボーディング、デバッグ、またはインシデントの調査の際に、コードベースの不慣れな領域についてチームが迅速にキャッチアップするのを支援します。

彼らはしばしば Codex を使用して、機能の中核ロジックを特定し、サービスまたはモジュール間の関係を整理し、システムを通じたデータフローを追跡します。また、そうでなければ生成に大きな手作業が必要になるような、アーキテクチャのパターンや不足しているドキュメントを浮き彫りにするのにも役立ちます。

インシデント対応の間、Codexは、コンポーネント間の相互作用を可視化したり、障害状態がシステム全体にどのように伝播していくかを追跡したりすることで、エンジニアが新しい領域に迅速に立ち上がれるよう支援します。

## ユースケース 2: リファクタリングと移行

## ユースケース 3: パフォーマンス最適化

Codex は、パフォーマンスのボトルネックを特定し、対処するために使用されます。

チューニングや信頼性向上の取り組みの際、エンジニアは Codex に対して、非効率なループ、冗長な操作、コストの高いクエリなどの、低速またはメモリ集約的なコードパスを分析するよう促し、最適化された代替案を提案させます。その結果として、効率と信頼性の両面で有意義な改善につながることがよくあります。

Codex は、リスクのあるまたは非推奨のパターンを特定して、それらが依然としてアクティブに使用されている場合に警告することで、コードの健全性を支援する用途にも用いられます。私たちのチームは、長期的なテックデットを削減し、回帰（リグレッション）を事前に防ぐために、これを活用しています。

## ユースケース4: テストカバレッジの改善

Codexはエンジニアがより速くテストを書くのを支援します。特に、カバレッジが薄い、または完全に欠けている箇所で効果を発揮します。

バグ修正やリファクタリングに取り組む際、エンジニアはしばしば Codex に対して、エッジケースや起こりやすい失敗パスをカバーするテスト案を提示するよう依頼します。新しいコードの場合、関数シグネチャと周辺のロジックに基づいて、ユニットテストまたはインテグレーションテストを生成できます。

## ユースケース 5: 開発速度の向上

Codexは、開発サイクルの開始と終了の両方を加速することで、チームがより速く進めるのを支援します。

新しい機能を開始するとき、エンジニアはそれを使ってボイラープレートをスキャフォールドします。つまり、フォルダー、モジュール、APIスタブを生成して、すべての部品を手作業で配線しなくても、動作可能なコードを素早く立ち上げられるようにします。

プロジェクトがリリースに近づくにつれ、Codexは、バグのトリアージ、最終段の実装ギャップの埋め合わせ、ロールアウト用スクリプト、テレメトリ用フック、または設定ファイルの生成といった、より小規模だが不可欠なタスクを処理することで、厳しい締め切りに対応するのに役立ちます。

また、プロダクトのフィードバックをスターターコードに変換するためにも使われます。エンジニアは多くの場合、ユーザーの要望や仕様を貼り付け、Codex にそれを基に、あとで返して改良できるようなラフドラフトを生成させます。

## ベストプラクティス

Codexは、構造、文脈、そして反復するための余地が与えられると、最も効果を発揮します。ここでは、OpenAIの各チームが日々の業務でそれから一貫した価値を得るために育成しているいくつかの習慣を紹介します。

### Askモードから開始してください

大きな変更の場合は、まず Ask mode を使用して Codex に実装計画をプロンプトします。これが、Code Mode に切り替えた後のフォローアッププロンプトの入力になります。この2段階のフローにより Codex は状況に即した状態に保たれ、出力の誤りを回避するのに役立ちます。Codex は、完了までにあなたやチームメイトが約1時間ほどかかるか、または実装に数百行のコードを要するような、適切に範囲設定されたタスクで最も効果を発揮します。モデルが改善されるにつれ、取り組めるタスクの規模が大きくなることを期待してください。

### Codexの開発環境を反復的に改善する

スタートアップスクリプト、環境変数、インターネットアクセスを設定することで、Codexのエラー率は大幅に低下します。タスクを実行する際は、Codexの環境設定で修正できるビルドエラーがないか確認してください。数回の反復が必要になる場合がありますが、長期的には大きな効率向上が得られます。

### GitHub Issueを書くようにプロンプトを構成してください

Codex は、プロンプトが PR や issue で変更を説明する方法をそのまま反映していると、より良い応答を返します。つまり、関連する場合はファイルパス、コンポーネント名、差分（diff）、ドキュメント抜粋を含めることです。たとえば「[module X] で行われているのと同じ方法でこれを実装して」のようなパターンでプロンプトを出すと、結果が向上します。

### Codex タスクキューを軽量なバックログとして使用します

関連するアイデア、途中までの作業、または偶発的な修正を拾い上げるためにタスクを投げてください。最初から一度で完全なPRを生成する必要はありません。Codexは、集中に戻ったときに再び戻ってこられるステージングエリアとしてうまく機能します。

### AGENTS.md を使用して永続的なコンテキストを提供します

プロンプト全体でリポジトリ内の Codex の動作をより効果的にするために、AGENTS.md ファイルを維持してください。これらのファイルには、通常、命名規則、業務ロジック、既知の癖、または Codex がコードのみからは推測できない依存関係が含まれます。ドキュメント内の AGENTS.md（‡1）ファイルの構成について詳しくは、（‡2）をご覧ください。

### 「Best of N」を活用して出力を改善する

Best-of-N 機能により、1つのタスクに対して複数の応答を同時に生成できるため、複数の解決策を素早く探索し、その中から最適なものを選べます。より複雑なタスクでは、いくつかの反復を確認し、異なる応答の一部を組み合わせて、より強力な結果を得られます。
## これから先

Codex はまだリサーチプレビューの段階ですが、すでに私たちの開発に実際の大きな影響を与えています。開発をより速く進め、より良いコードを書くことを助け、そうでなければ優先されることのなかった作業に取り組めるようになります。

私たちはこれからの可能性に胸を躍らせています。モデルがより良くなり、Codex が私たちのワークフローにより深く統合されていくにつれて、それを活用してソフトウェアを開発するさらに強力な方法を切り開けることを楽しみにしています。私たちは、その過程で学んだことを引き続き共有していきます。
